# AGENTS.md

## Назначение проекта

Это серверная часть консьерж-приложения: пользователи общаются с операторами
(консьержами) через чат. Проект находится на стадии MVP, но рассчитан на
дальнейшую разработку и последующий выход на рынок.

## Технологический стек

- Java 21.
- Maven Wrapper.
- Spring Boot.
- Spring Cloud Gateway.
- PostgreSQL для `auth-service`.
- MongoDB для `chat-service`.
- RabbitMQ STOMP relay для WebSocket/STOMP-сообщений.
- Docker Compose как основной способ запуска и деплоя.

## Структура проекта

- `auth-service/` - сервис регистрации, логина и выпуска JWT.
- `chat-service/` - сервис REST/WebSocket-чата, очереди пользователей и
  операторов, хранения сообщений и чат-сессий.
- `gateway/` - входная точка, маршрутизация REST/WebSocket и Swagger/OpenAPI.
- `docker-compose.yml` - основной сценарий запуска инфраструктуры и сервисов.
- `pom.xml` - корневой Maven POM для модулей проекта.

## Основной способ запуска

Основной сценарий разработки и деплоя - через Docker Compose:

```bash
docker compose up --build
```

Локальные Spring profiles пока не используются. Если они понадобятся, добавлять
их нужно отдельной задачей.

## Проверки перед завершением задачи

Если менялся код конкретного Maven-модуля, проверяй этот модуль:

```bash
./mvnw -pl <module-name> test
```

На Windows можно использовать:

```bash
mvnw.cmd -pl <module-name> test
```

Если изменение затрагивает Dockerfile, `docker-compose.yml`, маршрутизацию
gateway или межсервисные настройки, можно запускать Docker-команды для проверки.

Сейчас в проекте нет тестового покрытия, поэтому не требуй обязательного
добавления тестов к каждой задаче. Тестирование будет введено позже, когда MVP
станет ближе к готовому рабочему состоянию.

## Архитектурные правила

- `auth-service` отвечает за пользователей, роли, пароли и выпуск JWT.
- `chat-service` отвечает за чат, очередь, WebSocket/STOMP, хранение сессий и
  сообщений.
- `gateway` отвечает за внешний вход, маршруты и публичные Swagger/OpenAPI
  точки документации.
- Не закрепляй текущую схему общего JWT secret между сервисами как целевую
  архитектуру. Это временное MVP-решение.
- Операторы выходят "на линию", пользователи отправляют заявки на чат, и
  операторы получают пользователей. Возможен и обратный сценарий развития.
- Пользователь может писать сообщения в пустой чат, пока идет поиск оператора:
  никто не должен быть вынужден ждать, чтобы начать вводить сообщения.
- Роли `USER` и `EXECUTOR` не считать окончательными. В будущем могут появиться
  новые роли.

## Контракты API

Изменение любых внешних контрактов требует отдельного согласования с владельцем
проекта.

К контрактам относятся:

- REST paths и payloads.
- WebSocket endpoint `/ws`.
- STOMP destinations.
- Форматы сообщений и DTO.
- JWT claims, если они используются между сервисами или клиентом.
- Gateway routes.
- Swagger/OpenAPI paths.

Frontend существует и подстраивается под backend, но это не означает, что
контракты можно менять молча.

## Swagger/OpenAPI

На текущем этапе Swagger/OpenAPI через gateway считается публичной точкой
документации. При изменении REST API проверяй, что документация остается
актуальной.

## Стиль кода

- Следуй существующему стилю проекта.
- Используй Lombok там, где он уже применяется.
- Используй Java records для DTO, если это соответствует соседнему коду.
- Не добавляй новые абстракции без необходимости.
- Не выполняй крупные рефакторинги без отдельного согласования.
- Сохраняй текущую слоистую структуру: controller, service, repository, entity,
  dto, config.
- Не меняй версии Spring Boot, Spring Cloud и других ключевых зависимостей без
  подтверждения владельца проекта.

## Инфраструктура

Можно менять `docker-compose.yml`, Dockerfile и инфраструктурные настройки, если
это необходимо для задачи.

Секреты и пароли сейчас допустимы в конфигурационных файлах как часть MVP.
Перед production-этапом безопасность будет пересмотрена. При этом не добавляй
новые реальные production-секреты в репозиторий.

`ddl-auto: update` пока допустим для MVP. В production планируются миграции.
Не внедряй Flyway или Liquibase без отдельной задачи.

## Локализация и кодировка

Сообщения приложения могут быть на русском и английском.

Если встречается поврежденная кириллица или проблема кодировки, ее можно и
нужно исправлять при наличии уверенности в исходном смысле текста. Используй
UTF-8.

## Работа агента

- Перед изменениями смотри `git status`.
- Сначала изучай существующий код и локальные паттерны.
- Для поиска используй `rg`.
- Не откатывай чужие изменения без явного запроса.
- Не делай destructive-команды без явного разрешения.
- Не меняй контракты без согласования.
- Не выполняй крупные рефакторинги вместе с небольшими исправлениями.
- Если проверка не была запущена, явно укажи причину.
- В финальном ответе кратко описывай, что изменено и какие проверки выполнены.

## Известный технический долг

- `gateway` должен быть полноценным модулем в корневом `pom.xml`, но сейчас
  может отсутствовать в списке Maven modules.
- В проекте пока нет тестового покрытия.
- Секреты и пароли находятся в конфигурационных файлах как временное MVP-решение.
- Часть русскоязычных строк может быть повреждена из-за проблем кодировки.
- CI/CD пока не настроен; это запланировано на будущий этап.
- Для production понадобятся миграции БД вместо `ddl-auto: update`.
